# Implementation Plan: Ralph Loop Infrastructure

## Summary

Build and wire the Ralph Loop execution infrastructure into the harness scaffold. The harness references "The Ralph Loop Protocol" in governance files (AGENTS.md, QUALITY_SCORE.md, RELIABILITY.md, PLANS.md) but has no actual loop orchestration. This change makes the harness Ralph-loop-ready on day one: when someone copies `harness/` into their project, they get a complete end-to-end workflow — PRD creation, implementation planning, peer review, and autonomous loop execution — with all the plumbing pre-wired.

**User workflow after this change:**

1. `spec-prd-creator` — write PRD (already exists)
2. `spec-implementation-planner` — break PRD into increments (already exists, needs status fields)
3. `spec-peer-reviewer` — validate plan (already exists, add readiness lens)
4. `ralph-loop-runner` — execute the plan autonomously (new)

**Key design decisions:**

- **Workspace colocation**: During execution, PRD + exec plan + progress.txt live together in `docs/exec-plans/active/NNNN_feature-name/`
- **Status tracking via `**Status:** pending|done`** on increments and user stories — LLM-parseable, human-readable
- **One increment = one commit** — atomic, reversible, auditable
- **Archival on completion**: PRD → `docs/product-specs/`, exec plan → `docs/exec-plans/completed/`, progress.txt → deleted
- **`ralph.sh`** bash script for autonomous multi-iteration execution (fresh agent per increment)

Reference: No standalone PRD — originated from `AGENTS.md` Ralph Loop Protocol and `PLANS.md` Phase 2 gap analysis.

Principles:

1. One increment equals one coherent change set.
2. Every increment has an explicit test gate.
3. All changes within `harness/` — portable template self-containment preserved.
4. No `../` references from any file inside `harness/` (except template AGENTS.md files).
5. Existing skill behavior unchanged — status fields are additive only.

### Increment 04: Create ralph-loop-runner SKILL.md

**Status:** done

Scope:

1. Create skill folder `harness/workforce/agent-template/skills/ralph-loop-runner/`
2. Write `SKILL.md` with YAML metadata and full skill body covering:
   - **Overview**: What the Ralph Loop is, how it works
   - **Prerequisites**: PRD and exec plan must exist, peer review recommended
   - **Workspace Setup Protocol**: Create `docs/exec-plans/active/NNNN_feature-name/`, move PRD and exec plan into it, create `progress.txt`, create feature branch `ralph/NNNN-feature-name`
   - **Iteration Protocol**: Read workspace → find next `**Status:** pending` increment → set to `in-progress` → implement → run test gate → pottery-wheel if failing → mark `done` → update PRD checkboxes → commit → update progress.txt → exit (or continue to next)
   - **Marking Convention**: How to update status fields and PRD checkboxes
   - **Progress Log Format**: Structured format for progress.txt entries (iteration number, increment, status, files changed, learnings, commit hash)
   - **Commit Convention**: `feat|fix|refactor(NNNN): increment XX — title`
   - **Completion Detection**: All increments `**Status:** done` → run archival protocol
   - **Archival Protocol**: Move PRD back to `docs/product-specs/`, move exec plan to `docs/exec-plans/completed/`, delete progress.txt, remove empty workspace folder, optionally invoke `git-pr-creator`
   - **Error Recovery**: Blocked increments, crash recovery via git state, max 3 pottery-wheel retries per increment

Primary files:

1. `harness/workforce/agent-template/skills/ralph-loop-runner/SKILL.md` (new)

Test gate:

1. `npm run lint:md` passes
2. YAML frontmatter is valid
